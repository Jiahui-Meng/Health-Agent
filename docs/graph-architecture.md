# Health Graph Architecture / 健康图谱架构

## Overview

### English

This document describes the current health graph model used by Health Agent.

The graph is stored in the same relational database as the rest of the application. It is not backed by a dedicated graph database. The implementation is primarily defined in:

- `backend/app/models.py`
- `backend/app/services/graph_service.py`
- `backend/app/services/association_builder.py`
- `backend/app/services/association_analysis.py`
- `backend/app/schemas.py`

The graph has three layers:

- base graph: user, profile, session, symptom, timeline, summary, risk
- rule-derived association overlay: candidate links inferred from deterministic rules
- model-analysis overlay: manually triggered association links generated by the configured model

### 中文

本文说明 Health Agent 当前使用的健康图谱模型。

图谱与其他业务数据一起存储在同一个关系型数据库中，并不是独立的图数据库。主要实现位于：

- `backend/app/models.py`
- `backend/app/services/graph_service.py`
- `backend/app/services/association_builder.py`
- `backend/app/services/association_analysis.py`
- `backend/app/schemas.py`

当前图谱分为三层：

- 基础图层：用户、档案、会话、症状、时间线、总结、风险
- 规则派生关联层：基于确定性规则生成的候选关联边
- 模型分析关联层：手动触发后由当前模型生成并写回的关联边

## Storage Model

### English

The graph lives in two tables:

- `user_graph_nodes`
- `user_graph_edges`

These tables are defined in `backend/app/models.py`.

`user_graph_nodes` stores one node per row, with core fields:

- `id`
- `user_id`
- `node_type`
- `label`
- `payload` (JSON)
- `source`
- `created_at`
- `updated_at`

`user_graph_edges` stores one directed edge per row, with core fields:

- `id`
- `user_id`
- `from_node_id`
- `to_node_id`
- `edge_type`
- `payload` (JSON)
- `created_at`

## Root Shape

### English

The graph is user-centered.

At a high level, the structure is:

```text
user
  -> condition / medication / allergy
  -> session
       -> symptom
       -> symptom_event
       -> timeline_marker
       -> risk_signal
       -> summary
```

Important distinction:

- long-term features attach close to the `user` root
- visit-scoped evidence attaches under `session`
- association overlays connect existing nodes across these layers, but do not replace the base tree

### 中文

图谱以用户为中心。

整体结构如下：

```text
user
  -> condition / medication / allergy
  -> session
       -> symptom
       -> symptom_event
       -> timeline_marker
       -> risk_signal
       -> summary
```

关键区别：

- 长期特征挂在 `user` 根节点附近
- 单次问诊证据挂在 `session` 节点之下
- 关联分析层只是在这些节点之间补充连线，不会替代基础树结构

- `session`

- `risk_signal`

## Write Paths

### English

### User create or update

When a user is created or edited:

- ensure one `user` root node exists
- upsert long-term feature nodes:
  - `condition`
  - `medication`
  - `allergy`
- reconnect those nodes to the root

Long-term profile writes are replacement-based. Old profile-sourced nodes of the same type are removed, then normalized current values are inserted again.

### Chat write path

For each chat round:

1. ensure a `session` node exists
2. connect `user -> session` via `HAS_SESSION`
3. extract symptom labels and write `symptom` and `symptom_event`
4. extract time or evolution markers and write `timeline_marker`
5. write a `summary` node if assistant summary exists
6. write `risk_signal` only if the round is final:
   - `answer.stage == "conclusion"`
   - or emergency direct output

This prevents intermediate intake rounds from polluting final risk history.

### Rule association rebuild

Rule-derived association edges are rebuilt from current valid graph evidence. They can connect:

- long-term traits to session evidence
- repeated symptoms across sessions
- cycle-like repeated abdominal pain patterns for female users when timing evidence exists
- medication or allergy context to compatible symptom evidence

### Manual model association analysis

The graph page can trigger a full-user association analysis.

That flow:

1. gathers the user profile, current graph, valid sessions, valid messages, and existing rule-level associations
2. sends them to the configured model with a dedicated JSON-only prompt
3. validates returned references against existing node and session IDs
4. deletes previous `MODEL_*` edges for that user
5. writes the latest model-analysis edges back into `user_graph_edges`

### 中文

### 用户创建或更新

当用户被创建或编辑时：

- 确保存在一个 `user` 根节点
- upsert 长期特征节点：
  - `condition`
  - `medication`
  - `allergy`
- 重新把这些节点连接到根节点

长期档案写入采用替换式策略。旧的同类型 profile 节点会先删除，再写入新的规范化值。

### 聊天写入路径

每一轮聊天都会执行：

1. 确保存在一个 `session` 节点
2. 通过 `HAS_SESSION` 连接 `user -> session`
3. 抽取症状标签并写入 `symptom` 和 `symptom_event`
4. 抽取时间或病程变化标记并写入 `timeline_marker`
5. 如果 assistant 有总结，则写入 `summary` 节点
6. 只有在最终结论轮次才写入 `risk_signal`：
   - `answer.stage == "conclusion"`
   - 或急症直接输出

这样可以避免中间 intake 轮次污染最终风险历史。

### 规则关联重建

规则层候选关联边会基于当前仍然有效的图谱证据进行重建。它们可以连接：

- 长期特征与会话证据
- 跨会话重复症状
- 女性用户在存在时间模式证据时的周期性腹痛
- 用药或过敏上下文与兼容的症状证据

### 手动模型关联分析

图谱页可以手动触发一次全量关联分析。

这条流程会：

1. 收集用户资料、当前 graph、有效 sessions、有效 messages，以及当前规则层关联边
2. 用专门的 JSON-only prompt 发送给当前配置的模型
3. 校验模型返回的引用是否指向真实存在的节点和会话
4. 删除该用户旧的 `MODEL_*` 边
5. 将最新模型分析边重新写回 `user_graph_edges`

## Read Paths And Derived Summary

### English

The API returns both raw graph data and a derived summary bundle.

The main read helpers are:

- `get_graph_bundle(...)`
- `get_graph_payload(...)`

The returned summary bundle includes:

- `persistent_features`
- `profile_highlights`
- `recent_timeline`
- `recent_journey`
- `risk_signals`
- `summary_labels`

Meaning of these fields:

- `persistent_features`: grouped long-term traits such as conditions, medications, and allergies
- `profile_highlights`: lightweight user highlights such as birth year, sex, and region
- `recent_timeline`: raw timeline-like nodes for recent use
- `recent_journey`: readable symptom progression cards built from event history, assistant summaries, or session-level fallback text
- `risk_signals`: final risk markers sorted by current session first and higher severity first
- `summary_labels`: latest summary labels used as auxiliary context

The frontend graph page then projects raw `nodes + edges + summary_bundle` into a tree view.

That page keeps:

- the base tree as the default structure
- rule-derived association edges as an optional overlay
- model-analysis edges as another optional overlay

主要读取入口包括：

- `get_graph_bundle(...)`
- `get_graph_payload(...)`

返回的 `summary_bundle` 包括：

## Reconcile, Pruning, And Deletion

### English

The graph is designed to stay iteratively correct.

### Reconcile on read

When graph data is read, the system can reconcile missing derived data from still-existing sessions and messages. This helps old users recover graph content without a one-off migration script.

### Session deletion

Deleting a session removes more than rows from `sessions` and `messages`.

The cleanup also removes:

- the `session` node
- derived nodes whose `payload.session_id` points to that session
- related edges
- association edges whose `source_session_ids` reference the deleted session
- orphan nodes that no longer have valid evidence

### Invalid edge pruning

If an edge references a node or session that no longer exists, it is excluded from returned graph payloads. This applies to both rule-level associations and model-analysis edges.

### Stability detail

Derived children can be rebuilt, but stable session nodes are preserved when appropriate so overlays do not become unnecessarily fragile. The system treats association edges as a derived layer that must remain consistent with the current valid node set.

### 中文

图谱的设计目标是保持“可迭代且当前正确”。

### 读取时 reconcile

当读取 graph 数据时，系统会尝试根据当前仍存在的 sessions 和 messages 回填缺失的派生图数据。这样旧用户无需跑一次性迁移脚本，也能逐步恢复图谱内容。

### 删除会话

删除会话不只是删除 `sessions` 和 `messages` 表中的记录。

清理还会同步删除：

- `session` 节点
- `payload.session_id` 指向该会话的派生节点
- 相关边
- `source_session_ids` 引用了该会话的关联边
- 失去有效证据的孤儿节点

### 无效边 pruning

如果一条边引用了已经不存在的节点或会话，它不会出现在最终返回的 graph payload 中。这条规则同时适用于规则层关联边和模型分析边。

### 稳定性说明

派生子节点可以被重建，但稳定的 session 节点会在合适的情况下被保留，这样 overlay 不会因为无意义的节点重建而变得脆弱。系统将关联边视为一层派生语义，必须始终与当前有效节点集合保持一致。

## Iterative Update Strategy

### English

The graph follows this strategy:

- keep raw event evidence
- rebuild derived meaning from currently valid data
- separate factual storage edges from analysis overlays

In practice:

- raw symptom and timeline events accumulate per session
- long-term profile nodes are replaced or upserted
- derived summary views are rebuilt from valid graph state
- rule-level associations are rebuilt from current evidence
- model-analysis edges are refreshed only when the user manually runs full association analysis
- deleted sessions are removed physically, and their derived overlays are pruned

### 中文

图谱遵循的策略是：

- 保留原始事件证据
- 基于当前有效数据重建派生语义
- 将事实存储边与分析 overlay 分开管理

实际表现为：

- 原始症状事件和时间线事件会按 session 累积
- 长期档案节点采用替换或 upsert
- 派生摘要视图会从当前有效图状态重新生成
- 规则层关联边会根据当前证据重建
- 模型分析边只会在用户手动运行全量关联分析时刷新
- 被删除的 session 会被物理移除，与之相关的 overlay 也会一起被清理
